A tuned XGBoost model detected 437 additional fraudulent transactions missed by the baseline — while maintaining 97.2% precision.

Out of 644 fraudulent transactions in the test set, a simple Logistic Regression baseline caught only 59. A tuned XGBoost classifier — using behaviour-based features like spending anomalies and Card-Not-Present proxies — caught 496, at 97.2% precision.

Purpose

This project compares baseline and tuned machine-learning approaches for detecting credit card fraud in a severely imbalanced dataset (0.4% fraud rate), and explains which transaction features actually separate fraud from legitimate activity.

Why precision matters

A model can catch more fraud simply by flagging more transactions — but every false alert costs an investigator's time. High precision means fewer legitimate customers get incorrectly flagged, so fraud analysts spend their time on real cases instead of chasing false alarms.

00 — Key Insights

What the model comparison shows

  • 🎯Tuning XGBoost lifted precision from 93% to 97.2% while raising recall too — a rare win-win, not the usual trade-off.
  • 📉The Logistic Regression baseline missed 91% of all fraud in the test set — only 59 of 644 cases caught.
  • 💳Card-Not-Present transactions show a 3.44% fraud rate vs 0.02% for Point-of-Sale — over 170x higher risk.
  • 🕛Fraud peaks in the late night hours (11PM–1AM) and concentrates in online/CNP merchant categories.

02 — Method

Behaviour-based features, not just raw fields

Rather than feeding raw transaction fields straight into a classifier, the model is given engineered, interpretable behavioural signals — designed so a fraud analyst could understand *why* a transaction was flagged, not just that it was.

Most influential features (SHAP analysis, ranked)
  1. 01amt_diff_from_avg — how far a transaction's amount deviates from the cardholder's typical spend
  2. 02distance_to_merchant_km — geographic distance between cardholder and merchant location
  3. 03CNP_proxy — flag for likely Card-Not-Present (online/remote) transactions

Why this approach

  • XGBoost handles severe class imbalance (0.4% fraud) far better than linear models.
  • Engineered features (CNP proxy, spend deviation, distance) give interpretable signal, not black-box patterns.
  • GridSearchCV tuning + a 0.7 decision threshold prioritises precision, reducing false alarms for analysts.

Limitations

  • CNP/POS labels are heuristic proxies, not ground-truth — some misclassification is possible.
  • Static threshold; doesn't adapt per merchant category or in real time.
  • No real-time data streaming — trained and evaluated on static batch data.
  • Dataset lacks user demographics and device/browser metadata.
Why this matters

A 91%-miss-rate baseline isn't just a bad model — at scale, it's the difference between catching fraud and quietly absorbing it. Moving from a model that catches 1 in 10 cases to one that catches 8 in 10, at high precision, is the difference between a fraud team drowning in false alarms and one that can actually act on alerts.

03 — Deployment

From notebook to a working demo

The tuned XGBoost model was packaged into a live Gradio interface — users enter transaction amount, hour, weekend/night flags, and merchant distance, and get a real-time fraud probability score using the same GridSearch-optimised model, thresholded at 0.7 for high precision. The same model could be deployed behind a REST API and wired into a transaction-monitoring pipeline for real-time fraud scoring in production, rather than running as a standalone demo.
Production roadmap
  1. Deploy the model behind a live API or streaming dashboard for real-time alerts.
  2. Add a feedback loop so analyst-confirmed outcomes retrain the model over time.
  3. Layer in unsupervised anomaly detection to catch fraud patterns not seen in training data.
  4. Tune decision thresholds per merchant category rather than using one static cutoff.
Operational impact
  1. Fewer false alerts for investigators to manually review and clear.
  2. Detects over eight times as many fraudulent transactions as the baseline, at high precision.
  3. Analyst time redirected toward confirmed high-risk cases instead of false positives.
  4. A precision-first threshold reduces the risk of incorrectly blocking legitimate customers.
